1. Contexto y Objetivo del Proyecto

La organización necesita automatizar el proceso de recepción y validación de albaranes de proveedores en sus puntos de venta (tiendas). Actualmente, el proceso es manual y propenso a errores de transcripción, discrepancias con los pedidos y retrasos en la actualización del inventario en el ERP (Microsoft Dynamics 365 Business Central).

1.1 Objetivos

  • Automatizar la digitalización, extracción y validación de la información contenida en albaranes de múltiples proveedores con formatos heterogéneos.
  • Reducir el tiempo desde la recepción física del albarán hasta la actualización del inventario en Business Central.
  • Implementar un flujo de aprobación humana (Human-in-the-Loop) integrado con Microsoft Teams para gestionar discrepancias.
  • Garantizar trazabilidad extremo a extremo de cada albarán procesado.
  • Desplegar toda la solución en la nube de Microsoft Azure con una arquitectura basada en contenedores y agentes de IA.

1.2 Alcance

El alcance incluye:

  • Desarrollo de la pipeline completa de procesamiento de albaranes (captura → extracción → validación → inventario).
  • Tres agentes de IA orquestados mediante Microsoft Agent Framework v1.0+ y hospedados en Azure AI Foundry Agent Service.
  • Integración bidireccional con Dynamics 365 Business Central mediante MCP (Model Context Protocol).
  • Flujo de aprobación en Microsoft Teams.
  • Plataforma de observabilidad completa con Azure Monitor y Application Insights.
  • Infraestructura como Código (IaC) para despliegue reproducible en Azure Container Apps.

Pasos detallados

  1. Captura en tienda: El personal de tienda escanea o fotografía el albarán recibido del proveedor mediante una aplicación móvil o escáner conectado.
  2. Almacenamiento: La imagen se sube a un contenedor de Azure Blob Storage dedicado (albaranes-raw).
  3. Evento de disparo: La creación del blob genera un evento mediante Azure Event Grid que invoca al primer agente de IA.
  4. Agente 1 — Extracción: El agente procesa la imagen utilizando capacidades de visión (Azure AI Document Intelligence + modelo multimodal) y extrae campos estructurados en JSON.
  5. Persistencia: El JSON resultante se almacena en Azure Cosmos DB como un nuevo documento, generando a su vez un evento de Change Feed.
  6. Agente 2 — Validación: El Change Feed dispara al segundo agente, que obtiene el pedido original desde Business Central (vía MCP) y lo compara con los datos extraídos del albarán.
  7. Coincidencia → Agente 3: Si albarán y pedido coinciden, los datos pasan al Agente 3, que da de alta la mercancía en el inventario de Business Central (vía MCP).
  8. Discrepancia → Teams: Si hay diferencias, se envía una tarjeta adaptativa (Adaptive Card) al responsable de tienda en Microsoft Teams, detallando las discrepancias.
  9. Decisión humana: El responsable puede: aceptar las cantidades del albarán, corregir manualmente las cantidades, o rechazar el albarán.
  10. Resolución: Si se acepta (con o sin correcciones), se invoca al Agente 3 con los datos finales validados.

4. Orquestación Multi-Agente con Microsoft Agent Framework

4.1 Sobre Microsoft Agent Framework v1.0

Microsoft Agent Framework v1.0 (liberado en abril 2026) es el SDK de producción unificado que fusiona Semantic Kernel y AutoGen. Ofrece soporte oficial para .NET y Python, con patrones de orquestación multi-agente integrados, interoperabilidad MCP/A2A, y telemetría OpenTelemetry nativa.

4.2 Diseño de flujos — Análisis

Tras analizar el proceso completo, se recomienda descomponerlo en dos flujos principales orquestados y un sub-flujo condicional:

Flujo Patrón de orquestación Disparador Agentes involucrados
Flujo 1: Extracción y persistencia Event-Driven → Single Agent Event Grid (Blob creado) Agente 1 (Extracción)
Flujo 2: Validación y decisión Sequential Workflow con Handoff condicional Cosmos DB Change Feed Agente 2 (Validación) → Agente 3 (Inventario)
Sub-flujo 2b: Discrepancia + HITL Handoff Workflow + Human-in-the-Loop Resultado de Agente 2 (discrepancia) Agente 2 → Teams → Agente 3 (Inventario)

4.3 Justificación del diseño en dos flujos

¿Por qué dos flujos y no uno solo?
La separación en dos flujos desacoplados mediante Cosmos DB como capa intermedia ofrece:
  • Resiliencia: Si el Agente 2 o Business Central están caídos, los albaranes extraídos no se pierden; quedan en Cosmos DB y se reprocesan automáticamente.
  • Escalabilidad independiente: El Flujo 1 (OCR intensivo) puede escalar diferente al Flujo 2 (que depende de llamadas a BC).
  • Auditoría: Cosmos DB actúa como registro intermedio inmutable del dato extraído antes de cualquier decisión de negocio.
  • Reintentabilidad: Cada flujo puede reintentar de forma independiente sin re-ejecutar todo el pipeline.

4.4 Detalle de orquestación — Flujo 1 (Event-Driven Single Agent)

Blob creado → Event Grid → Webhook Container App → Agente 1: Extracción → Cosmos DB (Write) 

# Pseudocódigo — Flujo 1 con Microsoft Agent Framework (Python)
from agent_framework import Agent, FoundryChatClient
from agent_framework.tools import McpToolProvider

# Configurar herramientas MCP para el agente
blob_mcp = McpToolProvider(server="mcp-blob-storage")
cosmos_mcp = McpToolProvider(server="mcp-cosmos-db")
doc_intel_mcp = McpToolProvider(server="mcp-document-intelligence")

extraction_agent = Agent(
    client=FoundryChatClient(endpoint=FOUNDRY_ENDPOINT),
    name="ExtractionAgent",
    instructions="""Eres un agente experto en lectura de albaranes de entrega.
    Analiza la imagen proporcionada y extrae TODOS los campos relevantes
    en formato JSON estructurado: proveedor, fecha, nº albarán,
    líneas de detalle (producto, cantidad, unidad, precio), totales,
    observaciones, y referencia al pedido si existe.""",
    tools=[blob_mcp, doc_intel_mcp, cosmos_mcp]
)

# El webhook de Container App invoca:
async def handle_blob_event(event):
    blob_url = event["data"]["url"]
    result = await extraction_agent.run(
        f"Procesa el albarán ubicado en: {blob_url}"
    )
    # El agente guarda automáticamente en Cosmos via MCP tool
    return result

Cosmos Change Feed → Agente 2: Validación → ¿Coincide?

    SÍ → Handoff → Agente 3: Inventario

    NO → Teams HITL → Aprobado → Handoff → Agente 3 

# Pseudocódigo — Flujo 2 con Microsoft Agent Framework (Python)
from agent_framework import Agent, HandoffWorkflow, FoundryChatClient
from agent_framework.tools import McpToolProvider

bc_read_mcp = McpToolProvider(server="mcp-business-central-read")
bc_write_mcp = McpToolProvider(server="mcp-business-central-inventory")
cosmos_mcp = McpToolProvider(server="mcp-cosmos-db")
teams_mcp = McpToolProvider(server="mcp-teams-notifications")

validation_agent = Agent(
    client=FoundryChatClient(endpoint=FOUNDRY_ENDPOINT),
    name="ValidationAgent",
    instructions="""Comparas datos de albaranes extraídos con pedidos
    de Business Central. Identifica discrepancias en: cantidades,
    productos, precios, referencias. Emite un veredicto estructurado.""",
    tools=[cosmos_mcp, bc_read_mcp, teams_mcp]
)

inventory_agent = Agent(
    client=FoundryChatClient(endpoint=FOUNDRY_ENDPOINT),
    name="InventoryAgent",
    instructions="""Registras mercancía recibida en el inventario de
    Business Central. Solo realizas operaciones de ALTA (Create/Update).
    Nunca eliminas registros.""",
    tools=[bc_write_mcp, cosmos_mcp]
)

# Workflow con Handoff condicional
workflow = HandoffWorkflow(
    agents=[validation_agent, inventory_agent],
    handoff_strategy="conditional",
    # El validation_agent decide: handoff a inventory_agent
    # o escalación a human-in-the-loop
)

# Procesador de Change Feed
async def handle_change_feed(document):
    result = await workflow.run(
        f"Valida el albarán {document['id']} contra su pedido "
        f"referencia {document['ref_pedido']} y procede según resultado."
    )

5. Agente 1 — Extracción de Albaranes (OCR + IA)

5.1 Objetivo

Extraer de forma precisa y estructurada toda la información relevante de albaranes provenientes de múltiples proveedores con formatos heterogéneos (PDF, imagen, fax escaneado, etc.).

5.2 Estrategia de extracción (doble pasada)

Se recomienda una estrategia de doble pasada para maximizar la precisión:

  1. Pasada 1 — Azure AI Document Intelligence (v4.0): Extracción OCR de alto rendimiento usando modelos prebuilt (invoice) y/o modelos custom entrenados con albaranes representativos. Produce JSON estructurado con campos detectados, tablas y valores de confianza.
  2. Pasada 2 — Modelo multimodal (GPT-4o vía Azure OpenAI): El agente recibe tanto la imagen original como el JSON de la pasada 1. El LLM verifica, completa campos faltantes, resuelve ambigüedades y normaliza el formato de salida.
Se asume que el modelo prebuilt de invoice de Document Intelligence cubrirá la mayoría de formatos. Para proveedores con formatos muy atípicos, puede ser necesario entrenar un modelo custom neural con un mínimo de 5 documentos de ejemplo por formato. El proveedor deberá evaluar esta necesidad durante la fase de análisis.

6. Agente 2 — Validación contra Pedido (Business Central)

6.1 Objetivo

Comparar los datos extraídos del albarán con la información del pedido de compra original registrado en Dynamics 365 Business Central, e identificar de forma precisa cualquier discrepancia.

6.2 Lógica de validación

El agente debe realizar las siguientes comprobaciones:

Comprobación Fuente Albarán Fuente Pedido (BC) Tolerancia
Existencia del pedido ref_pedido Purchase Order Number Exacta
Proveedor correcto proveedor.nif Vendor No. Exacta
Productos incluidos lineas[].codigo_producto Purchase Lines → Item No. Exacta
Cantidades lineas[].cantidad Purchase Lines → Quantity Configurable (±5% por defecto)
Precios unitarios lineas[].precio_unitario Purchase Lines → Direct Unit Cost Configurable (±2% por defecto)
Se asume que los umbrales de tolerancia para cantidades (±5%) y precios (±2%) serán configurables mediante variables de entorno o un documento de configuración en Cosmos DB. Estos valores iniciales deben ser confirmados con el cliente.

8. Flujo Human-in-the-Loop (Microsoft Teams)

8.1 Descripción

Cuando el Agente 2 detecta discrepancias entre el albarán y el pedido, se activa un flujo de aprobación humana mediante una Adaptive Card enviada a Microsoft Teams al responsable de la tienda correspondiente.

8.2 Contenido de la Adaptive Card

La tarjeta debe incluir:

  • Cabecera: Nº de albarán, proveedor, fecha, referencia de pedido.
  • Tabla de discrepancias: Por cada discrepancia — producto, cantidad del albarán vs. cantidad del pedido, diferencia, impacto económico.
  • Resumen de impacto: Diferencia total en unidades y en importe.
  • Acciones disponibles:
    • ✅ Aceptar albarán — Se inventaría según las cantidades del albarán.
    • ✏️ Corregir cantidades — Campos editables para que el usuario ajuste las cantidades línea a línea antes de confirmar.
    • ❌ Rechazar albarán — Se marca como rechazado y no se procesa.

8.3 Flujo técnico

  1. El Agente 2 invoca la tool MCP de Teams para enviar la Adaptive Card al canal/chat correspondiente.
  2. La Adaptive Card incluye un Action.Submit que envía la respuesta a un webhook en Azure Container Apps.
  3. El webhook procesa la respuesta, actualiza el documento en Cosmos DB con la decisión y, si corresponde, invoca al Agente 3.
  4. Si no hay respuesta en un período configurable (timeout), se envía un recordatorio.
Se asume que existe un canal de Teams o chat grupal predefinido por tienda donde se enviarán las notificaciones. La resolución del destinatario (responsable de tienda) se obtendrá desde un maestro de tiendas configurable. Si no existe, el proveedor deberá proponer una solución de enrutamiento.
Se asume un timeout por defecto de 24 horas para la decisión humana, tras el cual se enviará un recordatorio. Si transcurren 48 horas sin respuesta, el albarán quedará en estado "pendiente_escalacion". Estos valores deben ser configurables.

8.4 Requisitos técnicos del HITL

Requisito Detalle

9. Integración MCP (Model Context Protocol)

9.1 Principio rector

Toda interacción de los agentes con servicios externos se realiza exclusivamente mediante MCP (Model Context Protocol). MCP actúa como la capa de abstracción universal que permite a los agentes descubrir y utilizar herramientas de forma estandarizada, segura y auditable.

9.2 Inventario de MCP Servers

9.4 Configuración MCP en Business Central

Dynamics 365 Business Central ofrece soporte nativo de MCP Server desde la wave 2 de 2025. La configuración requiere:

  1. Habilitar MCP Server en la configuración de Business Central.
  2. Definir API Pages expuestas: Seleccionar únicamente las entidades necesarias (Purchase Orders, Purchase Lines, Vendors, Items, Warehouse Receipts, Item Journal Lines).
  3. Configurar permisos por operación: Asignar Read/Create/Update según la tabla de permisos anterior. Desactivar Delete en todas las entidades.
  4. Autenticación OAuth 2.0: Configurar la aplicación registrada en Entra ID con los scopes mínimos necesarios.

11.1 Principio

Trazabilidad extremo a extremo obligatoria. Cada albarán procesado debe ser rastreable desde su captura hasta su alta en inventario (o rechazo), incluyendo cada decisión tomada por agentes y por humanos.

11.2 Stack de observabilidad

11.3 Integración con Azure AI Foundry

Azure AI Foundry Agent Service se integra nativamente con Application Insights mediante las convenciones semánticas GenAI de OpenTelemetry. La configuración requiere:

  1. Vincular un recurso de Application Insights al proyecto de Foundry en el portal Azure.
  2. Configurar el SDK de Microsoft Agent Framework con el exporter de OpenTelemetry para Application Insights.
  3. Habilitar la captura de spans para: invocación de agentes, llamadas a tools MCP, uso de tokens, resultados de evaluación.
# Configuración de telemetría con Agent Framework (Python)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from azure.monitor.opentelemetry.exporter import AzureMonitorTraceExporter

# Configurar exporter a Application Insights
exporter = AzureMonitorTraceExporter(
    connection_string="InstrumentationKey=xxxxx;IngestionEndpoint=https://..."
)
provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(exporter))
trace.set_tracer_provider(provider)

# El Agent Framework genera automáticamente spans para:
# - Cada invocación de agente
# - Cada llamada a tool MCP
# - Cada request/response al modelo LLM
# - Token usage (prompt + completion)

11.5 Trazabilidad de albarán (Correlation ID)

Cada albarán procesado debe tener un Correlation ID único (albaran_id) que se propague como trace_id de OpenTelemetry a través de todos los spans generados por los tres agentes, los MCP servers y el flujo HITL. Esto permite reconstruir la historia completa de un albarán en Application Insights mediante una sola consulta KQL: 

// KQL — Trazabilidad completa de un albarán
traces
| where customDimensions.albaran_id == "ALB-2026-00451"
| order by timestamp asc
| project timestamp, operationName, customDimensions.agent_name,
          customDimensions.mcp_server, customDimensions.estado, duration

12. Seguridad y Gobernanza

12.1 Identidad y acceso

  • Toda autenticación entre servicios se realiza mediante Managed Identities (System Assigned) de Microsoft Entra ID. Prohibido el uso de claves compartidas o connection strings en código.
  • Los MCP Servers de Business Central se autentican vía OAuth 2.0 Client Credentials Flow con scopes restringidos.
  • El acceso humano al portal de administración requiere MFA y roles RBAC de Azure.

12.2 Red y aislamiento

  • Azure Container Apps desplegado en un Virtual Network dedicado.
  • Cosmos DB y Blob Storage accesibles solo mediante Private Endpoints.
  • Key Vault accesible solo desde la VNet.
  • Todo el tráfico externo (Teams, BC) sale por un NAT Gateway con IP fija si se requiere allowlisting.

12.3 Gestión de secretos

  • Todas las claves, tokens y connection strings se almacenan en Azure Key Vault.
  • Container Apps referencia secretos de Key Vault directamente (sin copiarlos a variables de entorno).
  • Rotación automática de secretos habilitada.

12.4 Protección de datos

  • Cifrado en reposo: AES-256 (por defecto en todos los servicios Azure).
  • Cifrado en tránsito: TLS 1.2+ obligatorio.
  • Retención de imágenes de albaranes sujeta a la política de retención del cliente (configurable mediante lifecycle management de Blob Storage).
Se asume que no hay requisitos específicos de residencia de datos más allá de desplegar en una región Azure de la UE. Si existen requisitos GDPR específicos respecto a datos personales en albaranes (ej. nombres de transportistas), el proveedor deberá implementar mecanismos de anonimización.

13.1 Azure Container Apps

15. Suposiciones y Decisiones Abiertas

Las siguientes suposiciones se han tomado en ausencia de confirmación explícita. Deben ser validadas con el cliente antes del inicio del desarrollo:

# Suposición Impacto si es incorrecta
S1 La aplicación de captura en tienda (móvil/escáner) ya existe o será proporcionada por el cliente. Este pliego no incluye su desarrollo. Alto — Requeriría un módulo adicional de frontend/mobile.
S2 Business Central está en versión 2025 wave 2 o posterior, con soporte nativo de MCP Server. Alto — Si es una versión anterior, será necesario desarrollar una API custom como intermediario.
S3 Los pedidos de compra en Business Central siempre tienen un número de referencia que coincide o es mapeado con el campo ref_pedido del albarán. Medio — Se requeriría lógica de fuzzy matching o búsqueda por proveedor + fecha.
S4 Todos los albaranes incluyen una referencia al pedido de compra. Si algún albarán no la incluye, se derivará automáticamente al flujo HITL. Bajo — Ya cubierto por la lógica de discrepancia.
S5 El tenant de Microsoft 365 del cliente permite la instalación de bots personalizados en Teams. Alto — Impediría el flujo HITL y requeriría un canal alternativo (email, app web).
S6 Los tolerancias de validación (±5% cantidad, ±2% precio) son valores iniciales razonables. Bajo — Son configurables.
S7 El modelo LLM utilizado será GPT-4o o GPT-4.1 desplegado en Azure OpenAI Service dentro del mismo tenant. Bajo — Agent Framework soporta múltiples proveedores.
S8 No se requiere procesamiento offline (sin conectividad) en las tiendas. Alto — Requeriría arquitectura edge con buffer local.
S9 El volumen estimado es de hasta 500 albaranes/día en total (todas las tiendas). Medio — Volúmenes mayores podrían requerir ajustes de throughput y scaling.
S10 Se despliega en región de Azure en la UE sin requisitos de soberanía de datos adicionales. Medio — Regulaciones específicas podrían limitar servicios disponibles.
S11 El lenguaje de implementación principal será Python (alineado con el SDK principal de Agent Framework). Bajo — También disponible en .NET si el equipo lo prefiere.
S12 La interacción con Business Central para inventario se mapea a entidades estándar (Warehouse Receipt, Item Journal). No se requiere lógica de negocio custom en BC (ej. codeunits). Medio — Si existe lógica custom, podría requerir desarrollo en AL (Business Central).
